# 数据字典 · Dashboard 上每个数字都是怎么算出来的

> 给接手数据分析的同事看的。你只要会用 Excel 透视表 / pandas，就能 1:1 复现 Dashboard 上的所有数字。**所有计算都基于一份 CSV/XLSX 导出，不依赖外部数据**。

---

## 1. 用了 CSV/XLSX 里的哪些列

Agora ConvoAI Console 默认导出格式有十几列，但我只用以下这 8 列。其他列（Campaign ID / From Number / To Number / Channel 等）一律忽略。

| 列名 | 这里怎么用 |
|---|---|
| `Call ID` | 通话主键。导出时每行的唯一标识 |
| `Agent ID` | Agent 唯一标识 |
| `Agent Name` | 批次/版本名。Dashboard 顶部下拉框就是按这个分组的；空的填 `"(unknown)"` |
| `Duration (seconds)` | 通话总时长（整数秒）。用于 Duration 分布图、首句挂断 Duration 分布。空值当 0 处理 |
| `Transcript` | 一段 JSON 字符串，是一个数组。每个元素是一轮发言，含 `turn_id` / `role` (`assistant` 或 `user`) / `content`。**所有"轮次"相关的指标都从它数** |
| `Structured Output` | 一段 JSON 字符串，是一个对象，6 个字段：购车品牌 / 购车型号 / 购车城市 / 购车姓名 / 购车意向 / 购车时间。每个字段值是字符串或 `null`。**"完整转换"和"意向客户"都从它算** |
| `Hangup Reason` | 通话怎么结束的，是一个枚举字符串（见 §3 表）。**"真人接听"用它判** |
| `Audio Record File Download URL` | 录音 URL。只在导出 zip 时用，不算任何指标 |

## 2. Funnel 五层数字怎么来

```
拨打总数 → 真人接听 → 完整转换 → 带车型完整转换 → 意向客户
```

注意：**后三层是并列分支，不是嵌套**。一通通话可以同时是「完整转换」和「意向客户」，也可以只满足其中一个。Funnel 图为了视觉只是按数值从大到小排，语义上它们都从「真人接听」里独立切出来。

| 层 | 怎么数 |
|---|---|
| **拨打总数** | CSV 总行数。一行 = 一通 |
| **真人接听** | `Hangup Reason` 等于 `USER_HANGUP` 或 `AI_HANGUP` 的行数 |
| **完整转换** | "结构化收到了 4 个槽位中的至少 3 个"——详见 §4 |
| **带车型完整转换** | 是「完整转换」+ 槽位里包含「车型」（详见 §4） |
| **意向客户** | `Structured Output` 这一列 JSON 里 `"购车意向": "是"`。**注意只算字面值"是"**，"否" / null / "考虑中" 都不算 |

### 关于「接听」层

早期版本有「接听 = Duration > 0」层，但实测每批 接听 ≈ 拨打（99%+），看不出问题，移除。

## 3. `Hangup Reason` 枚举值 — 哪些算「真人接听」

| 值 | 算真人接听？ | 说明 |
|---|---|---|
| `USER_HANGUP` | ✅ | 用户主动挂断（最明确的真人信号） |
| `AI_HANGUP` | ✅ | AI 主动挂断（任务完成或对话自然结束。注意 AI 挂的前提一定是用户在线对话过） |
| `AI_ASSISTANT_HANGUP` | ❌ | 客户端在 agent 还在说时挂掉。常见于客户切话——但因为 agent 没把话讲完，划归非真人接听更稳妥 |
| `SILENCE_HANGUP` | ❌ | 静默超时——一直没人说话，断了 |
| `CALL_TIMEOUT` | ❌ | 系统超时 |
| `VOICE_MAIL` | ❌ | 转语音信箱 |
| `NO_ANSWER` | ❌ | 没接 |
| `BUSY` | ❌ | 占线 |
| `TEMPORARY_FAILURE` | ❌ | 链路故障 |

所以在 pandas 里就是：

```
真人接听数 = df[df["Hangup Reason"].isin(["USER_HANGUP", "AI_HANGUP"])]
```

## 4. `Structured Output` 字段和「4 槽位」

### 4.1 原始字段长这样

Agent 把通话里收集到的信息填进一个 JSON 对象（不填的字段是 `null`）：

```json
{
  "购车品牌": "奔驰",
  "购车型号": null,
  "购车城市": "广州",
  "购车姓名": null,
  "购车意向": "是",
  "购车时间": "1个月"
}
```

### 4.2 4 个「槽位」（slot）的判定

「完整转换」不是看 6 个字段，而是看以下 **4 个槽位**有几个填齐：

| 槽位 | 满足条件 |
|---|---|
| **车型** | `购车品牌` **或** `购车型号` 任一非 null 非空 |
| **时间** | `购车时间` 非 null 非空 |
| **城市** | `购车城市` 非 null 非空 |
| **姓名** | `购车姓名` 非 null 非空 |

**几个重要决策：**

- **「车型」槽合并了品牌和型号**——只要客户告诉了 agent 其中一个（比如只说"奔驰"或只说"E300"），就算填齐。理由：4S 店拿到任一都能跟进。
- **`购车意向` 不算槽位**——它是单独的"客户意向标签"，对应 funnel 的「意向客户」层，不计入"完整转换"。
- **空字符串 `""` 算作 null**——很多时候导出脚本把空值写成空串而非 JSON null，所以都当成"没填"处理。

### 4.3 怎么算「完整转换」

填齐的槽位数 ≥ 3 就算。

### 4.4 怎么算「带车型完整转换」

是「完整转换」**且**填齐的槽位里**包含「车型」**。语义上：客户至少告诉了 agent 想买什么车 + 至少其它 2 项硬下限。这一层才是真正能转给 4S 店的销售线索。

### 4.5 几个真实例子

| Structured Output | 车型? | 时间? | 城市? | 姓名? | 槽位数 | 完整转换? | 带车型完整? | 意向? |
|---|---|---|---|---|---|---|---|---|
| `{品牌:奔驰, 型号:null, 城市:广州, 姓名:null, 意向:是, 时间:1月}` | ✅(品牌) | ✅ | ✅ | ❌ | 3 | ✅ | ✅ | ✅ |
| `{品牌:null, 型号:E300, 城市:null, 姓名:张, 意向:是, 时间:null}` | ✅(型号) | ❌ | ❌ | ✅ | 2 | ❌ | ❌ | ✅ |
| `{品牌:路虎, 型号:极光, 城市:上海, 姓名:李, 意向:是, 时间:半年}` | ✅ | ✅ | ✅ | ✅ | 4 | ✅ | ✅ | ✅ |
| `{品牌:null, 型号:null, 城市:北京, 姓名:王, 意向:否, 时间:1月}` | ❌ | ✅ | ✅ | ✅ | 3 | ✅ | **❌**（缺车型） | ❌（意向="否"） |
| `{全部 null, 意向:是}` | ❌ | ❌ | ❌ | ❌ | 0 | ❌ | ❌ | ✅ |

### 4.6 为什么「意向」用 `=="是"` 而不是"非 null"

我们试过用"非 null"。看真实数据：

| `购车意向` 字段值 | 占真人接听 |
|---|---|
| `null` / 空（agent 没问到） | 73-95% |
| `"否"`（客户明确拒绝） | 5-22% |
| `"是"`（客户明确说要买） | 3-5% |

如果用"非 null"，会把"否"算进意向客户，那意向数会从 34 一下子涨到 188（5/14 第一波这批的实际数据），还把明确拒绝的客户当成有意向，逻辑反了。所以坚持 `==`"是"`。

LLM 真伪分析（§6）还会进一步把"是"再细分成 真意向 / 假意向 / 模糊。

## 5. 每张图表用了哪些数据

### 5.1 顶部 5 张 KPI 卡

5 个数字直接对应 Funnel 5 层。每张卡下方的百分比按以下口径：

| 卡 | 显示哪些百分比 |
|---|---|
| 拨打总数 | 不显示 |
| 真人接听 | 占总 |
| 完整转换 | 占总 · 占真人 |
| 带车型完整转换 | 占总 · 占真人 · **占完整** |
| 意向客户 | 占总 · 占真人 |

### 5.2 漏斗图

直接画 5 层数字，从大到小。点任一层 → 弹窗导出该子集的 Excel/录音。

### 5.3 「轮次分布」3 张子图

只看真人接听的通话。横轴是该通的 `max(turn_id)`（也就是这通最后一条发言的 `turn_id`，包含 agent 和真人双方）。三张子图分别展示：

- **真人接听 (全部)**：所有 `Hangup ∈ {USER, AI}_HANGUP` 的通
- **完整转换**：上面 ∩ 完整转换
- **意向客户**：上面 ∩ 意向客户

三张图 X 轴范围一致（全数据集的最大 max_turn_id），方便横向比。柱子上方百分比是相对该子图自己的总数。

### 5.4 Duration 分布

- 数据：真人接听通话的 `Duration (seconds)`
- X 轴：每秒一柱，从 0 到该批最长秒数
- 底部有拖动条，可缩放查看任意区间

### 5.5 「完整转换槽位分布」+ 下钻

**柱图** 横轴是「该通真人接听里填了几个槽位」（0/1/2/3/4），纵轴是通话数。

**下钻面板** 展示：

- 「4/4 全齐」=「完整转换 ∩ 4 槽位都填」的通话数
- 「仅 3/4」=「完整转换 ∩ 恰好 3 槽位填」的通话数
- 在「仅 3/4」内，缺哪个槽位的分布（缺车型 / 缺时间 / 缺城市 / 缺姓名）

下钻的"缺槽位"统计意义：能看出 agent 最不擅长收哪一项硬下限。

### 5.6 早期挂断表

只看真人接听里 agent 说话轮数（也就是 `Transcript` 里 `role == "assistant"` 的条目数）的分布：

| 桶 | 条件 |
|---|---|
| 首句挂断 (1 句) | agent 说话轮数 **正好等于 1** |
| 2 句挂断 | agent 说话轮数 **正好等于 2** |
| 3 句挂断 | agent 说话轮数 **正好等于 3** |
| 4 句挂断 | agent 说话轮数 **正好等于 4** |
| 5 句挂断 | agent 说话轮数 **正好等于 5** |

**互斥分桶**——2 句挂断不包括 1 句挂断的。

### 5.7 首句挂断 Duration 分布

只看「真人接听 ∩ agent 只说了 1 句」这批通话的 `Duration (seconds)`。柱图，每秒一柱。

切换两个视角：
- **全部**：所有首句挂断
- **短挂断 (<15 秒)**：进一步筛 `Duration < 15`，聚焦那些 AI 第一句还没说完就被切话的

## 6. LLM 真伪意向分析

意向客户层（购车意向="是"）的通话里，有些其实是客户客气敷衍（"考虑考虑" / "好的"）。这步把每通送给大模型再判断一次。

### 6.1 输入
- 对象：所有 `购车意向 == "是"` 的通话（一般每批几十通）
- 给大模型看的内容：
  - 完整 transcript（渲染成 `agent: ...\nuser: ...` 多行文本，超 3000 字符截断）
  - 完整 Structured Output JSON

### 6.2 大模型给出的判定

每通输出 3 个字段（JSON）：

| 字段 | 取值 | 含义 |
|---|---|---|
| `verdict` | 真意向 / 假意向 / 模糊 | 总体判定 |
| `reason` | ≤30 字 | 判断依据，比如「客户主动说品牌+城市+尾号」 |
| `evidence` | ≤50 字 | 客户原话片段 |

### 6.3 判定标准（也就是 prompt 里给大模型的规则）

- **真意向**：客户主动给具体信息（品牌/车型/时间/城市/预算/手机尾号），或主动问价格/优惠/4S 店地址，或明确同意接 4S 店回电
- **假意向**：客户只有"好的""考虑一下""有需要会联系"这种敷衍话术，没有具体动作
- **模糊**：既没明显敷衍也没明确承诺

### 6.4 模型 + 速度
- 默认 `qwen3.6-plus` (DashScope)
- 服务端 16 并发，单通约 1-5 秒
- 30 通约 90 秒，70 通约 2-3 分钟

## 7. 导出 Excel 的列含义

任意位置点导出，xlsx 都含这 12 列（一行 = 一通）：

| 列名 | 来源 |
|---|---|
| Call ID | 原始 |
| Agent ID | 原始 |
| Agent Name | 原始 |
| Duration (s) | 原始 `Duration (seconds)` |
| Hangup Reason | 原始 |
| Max turn_id | 该通 `Transcript` 里最大的 `turn_id` |
| Assistant turns | 该通 `Transcript` 里 `role==assistant` 的条目数 |
| Is Human Answered | True/False，按 §3 的规则 |
| Is Full Conversion | True/False，按 §4 的规则 |
| Is Intent | True/False，`购车意向=="是"` |
| Transcript | 把原始 JSON 渲染成 `role: content` 多行可读文本 |
| Audio URL | 原始 `Audio Record File Download URL` |

LLM 分析结果的 xlsx 多 4 列：`LLM Verdict / LLM Reason / LLM Evidence / LLM Error`。

## 8. 已知的坑 / 注意点

- **`Structured Output` 解析失败**：列里可能是空串/格式不对，会被当作"没结构化数据"。该通的「完整转换」和「意向」都是 False
- **`Agent Name` 缺失**：填 `"(unknown)"`，下拉框里能看到这一组
- **图表 X 轴跨 Agent 对齐**：max_turn_id 和 Duration 的范围都用整批最大值，切换 Agent 时 X 轴不动，方便横向比较
- **录音 URL 有效期 7 天**：是 Aliyun OSS 签名链接，旧批次跑分析就拉不到录音了，但其它指标不受影响
- **`Transcript` 为空/`[]`**：通常是没接通的通话。`assistant_turns` 和 `max_turn_id` 都是 0，所以早期挂断里不会出现
- **意向客户和完整转换并不嵌套**：一通可以"意向=是"但只填了 1 个槽位（非完整转换），或 4 槽位都填了但意向=否

## 9. 用 pandas / SQL 复现

如果你想在自己的环境里跑同一套指标，骨架是：

```python
import pandas as pd, json

df = pd.read_csv("summary-xxx.csv")

def parse_structured(s):
    try: return json.loads(s) if s and str(s).strip() else None
    except: return None

def filled_slots(d):
    if not d: return set()
    out = set()
    if d.get("购车品牌") or d.get("购车型号"): out.add("车型")
    if d.get("购车时间"): out.add("时间")
    if d.get("购车城市"): out.add("城市")
    if d.get("购车姓名"): out.add("姓名")
    return out

df["_so"]      = df["Structured Output"].apply(parse_structured)
df["_slots"]   = df["_so"].apply(filled_slots)
df["_human"]   = df["Hangup Reason"].isin(["USER_HANGUP","AI_HANGUP"])
df["_full"]    = df["_slots"].apply(lambda s: len(s) >= 3)
df["_full_wm"] = df["_full"] & df["_slots"].apply(lambda s: "车型" in s)
df["_intent"]  = df["_so"].apply(lambda d: d and d.get("购车意向") == "是")

# Funnel
print("拨打总数", len(df))
print("真人接听", df["_human"].sum())
print("完整转换", df["_full"].sum())
print("带车型完整转换", df["_full_wm"].sum())
print("意向客户", df["_intent"].sum())
```

跑出来的 5 个数字应该和 Dashboard 顶部 5 张卡片完全一致。

## 10. Tab 2 · Agent 视角 KDA（独立口径，不影响 Tab 1）

**Dashboard 顶部多了一个 tab**。Tab 2 用一套独立的关卡定义评估每个 Agent Name，不动 Tab 1。

### 10.1 4 关字段映射

| 关 | 通关条件 |
|---|---|
| 第 1 关 车型 | `购车品牌` **AND** `购车型号` 都非 null/非空（**与 Tab 1 完整转换的 OR 口径不同**）|
| 第 2 关 城市 | `购车城市` 非 null |
| 第 3 关 时间 | `购车时间` 非 null |
| 第 4 关 姓氏 | `购车姓名` 非 null |

严格线性递进：**过第 N 关 ⇔ 第 1..N 全过**。第 1 关没过就过不了 2-4 关。

### 10.2 产品视角 7 层漏斗

```
拨打总数 → 接听 (Duration>0) → 真人接听 (USER/AI_HANGUP)
         → 过车型关 (L1)
         → 过城市关 (1..2 全过)
         → 过时间关 (1..3 全过)
         → 全关通过 (1..4 全过)
```

### 10.3 6 维度雷达图（每个 Agent 一张）

所有维度归一化到 0-100，都在该 Agent 的"真人接听"范围内计算。

| # | 维度 | 公式 | 数据来源 |
|---|---|---|---|
| 1 | **击穿率** | `n_full / n_human × 100`（n_full = 4 关全过通话数）| Structured Output 4 字段 |
| 2 | **轮效** | 全过通话的平均 `max(turn_id)`：5 轮=100 分，15 轮=0，clip 到 [0,100] | Transcript |
| 3 | **首杀** | `100 − T1_avg × 15 − T2_avg × 8`，T1/T2 = 客户首次说出车型/城市关键词所在 turn_id。**v0 用关键词正则**，v1 上 LLM | Transcript |
| 4 | **滑顺** | `100 − friction × 20`。friction = "同一关 assistant 重复问 ≥ 2 次" 计数（4 关各自一票） | Transcript |
| 5 | **不偏科** | `100 − var(P1,P2,P3,P4) × 1000`，P_N = 第 N 关独立通过率（不要求线性递进） | Structured Output |
| 6 | **抗挂** | `(1 - 早挂断率) × 100`，早挂断 = assistant 轮数 ≤ 3 的通话 | Transcript |

**综合分** = `击穿率 × 0.3 + 轮效 × 0.2 + 首杀 × 0.15 + 滑顺 × 0.15 + 不偏科 × 0.1 + 抗挂 × 0.1`

### 10.4 Tab 1 vs Tab 2 口径冲突说明

| 项 | Tab 1 完整转换 | Tab 2 第 1 关车型 |
|---|---|---|
| 车型槽逻辑 | `购车品牌` OR `购车型号` | `购车品牌` AND `购车型号` |
| 触发条件 | ≥ 3 of 4 槽位填齐 | 严格线性，第 1 关必须最先过 |
| 业务含义 | 给 4S 店的销售线索 (品牌单独也算) | 给销售员打分 (信息完整度更严) |

故意分开，不要互相对齐。同一通通话在两个 tab 里可能"完整转换=是但车型关=未过"，是正常的。

### 10.5 关键词词典（v0 近似）

维度 3「首杀」和维度 4「滑顺」用关键词正则做粗略抽取。词典内置在 `scripts/lib/agent_kda.py` 的 `LEVEL_KEYWORDS_USER` / `LEVEL_KEYWORDS_ASSISTANT`。包含主流汽车品牌（宝马 / 奔驰 / 丰田…）、一线/省会城市（上海/北京…）、时间词（月/年底/近期…）、姓氏标识（"姓"/"我姓"/"先生"…）。**v0 漏报会偏高**，比如说"我看个 SUV"没有具体品牌就不算车型命中。v1 会换 LLM 抽取。
